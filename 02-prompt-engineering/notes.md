# 提示词工程 Prompt Engineering

> 对于强模型（Claude 3.5+），很多技巧已经不需要了。
> 但对于老模型（Gemini 3.0、GPT-3.5 级别），这些依然是必修课。

---

## 核心框架：RCTF

一个好的 prompt 通常包含四个要素：

```
R — Role       角色：你是谁（给 AI 设定身份）
C — Context    背景：任务的上下文信息
T — Task       任务：你要 AI 做什么（越具体越好）
F — Format     格式：你希望输出是什么样的
```

**示例对比：**

❌ 差的 prompt：
```
帮我写一封邮件给客户
```

✅ 好的 prompt（RCTF 结构）：
```
[Role] 你是一名专业的客户成功经理，语气友好但专业。

[Context] 我们的产品上周发生了一次服务中断，影响了约 200 名用户，
持续时间约 3 小时。问题已经修复，根本原因是数据库配置错误。

[Task] 写一封道歉邮件发给受影响的客户，说明发生了什么、
我们如何解决的、以及未来如何防止类似问题。

[Format] 邮件格式，200 字以内，语气诚恳，结尾提供补偿方案（一个月免费）。
```

## 技巧一：Chain-of-Thought（思维链）

**适用场景：** 复杂推理、分析类任务，老模型尤其有效。

**原理：** 让模型先"思考"再回答，而不是直接给结论。

**用法：**
```
分析这段代码有什么安全漏洞。请先逐行说明你的分析过程，再给出结论。
```

```
请一步一步思考：如果我们把用户数据库从 MySQL 迁移到 PostgreSQL，
会有哪些风险？
```

**为什么有效：** 模型在生成答案时，前面的"思考过程"会影响后面的结论。让它先写思考过程，等于强迫它更认真地推理。

---

## 技巧二：Few-shot Examples（举例示范）

**适用场景：** 需要特定输出格式或风格时，老模型几乎必用。

**原理：** 给模型看 1-3 个"输入→输出"的例子，它会模仿这个模式。

**用法：**
```
把下面的用户反馈分类为：正面 / 负面 / 中性

例子：
输入：这个产品太好用了！
输出：正面

输入：还行吧，没什么特别的
输出：中性

输入：客服态度很差，等了一个小时都没解决问题
输出：负面

现在请分类：
输入：界面设计挺漂亮的，但加载速度有点慢
输出：
```

**关键：** 例子要覆盖边界情况，不要全是简单的典型案例。

---

## 技巧三：约束条件（告诉 AI 不要做什么）

**适用场景：** 当模型总是做你不想要的事时。

老模型尤其喜欢：加免责声明、说"当然可以"之类的废话、回答跑题。

**用法：**
```
总结这篇文章的核心论点，100 字以内。
不要加任何前置说明，直接给出总结。
不要用"这篇文章"开头。
```

```
帮我 review 这段代码。
只指出真正的问题，不要夸奖好的地方，不要说"总体来说写得不错"。
```

---

## 技巧四：输出格式控制

**适用场景：** 需要结构化输出时（JSON、表格、列表等）。

老模型在格式控制上不如新模型稳定，需要更明确地指定。

**用法：**

要 JSON：
```
提取以下文本中所有人名和职位，以 JSON 格式返回：
{"people": [{"name": "...", "role": "..."}]}

文本：张总裁昨天在会议上宣布，市场总监王明将负责新产品线。
```

要表格：
```
对比以下三个方案的优缺点，用 Markdown 表格格式，
列：方案名 / 优点 / 缺点 / 适用场景
```

---

## 技巧五：角色设定的正确用法

**适用场景：** 需要特定专业视角或语气时。

**有效的角色设定：**
```
你是一个有 10 年经验的 Python 工程师，
对性能优化和代码可读性同样重视。
```

**没用的角色设定（太模糊）：**
```
你是一个专家。
```

**注意：** 强模型不太需要这个技巧，但 Gemini 3.0 级别的模型角色设定效果明显。

---

## 技巧六：迭代而非重来

**最重要的习惯，很多人不会用。**

拿到不满意的结果后，不要重新发一个 prompt，而是在当前对话基础上精炼：

❌ 差的做法：
```
（重新开一个对话）
写一封更正式的邮件...
```

✅ 好的做法：
```
（在同一对话里）
刚才那封邮件太口语化了，请改得更正式一些，
同时把第二段的道歉语气加强。
```

**为什么：** 模型有之前的上下文，知道你指的是什么。从零开始会损失很多信息。

## 常见错误

| 错误 | 为什么错 | 怎么改 |
|------|---------|--------|
| prompt 太短太模糊 | 模型猜你的意图，结果随机 | 加背景、加约束、加格式要求 |
| 期望一次完美 | 没有 AI 能一次做对所有事 | 先要草稿，再迭代 |
| 不说格式要求 | 模型自由发挥，你不喜欢 | 明确说"用列表/表格/JSON" |
| 一个 prompt 塞太多任务 | 模型容易遗漏或混乱 | 拆开成多个步骤 |
| 拿到坏结果就放弃 | 错过了迭代改进的机会 | 告诉它哪里不对，为什么 |

## 给老模型的额外提示

使用 Gemini 3.0 / GPT-3.5 级别的模型时：
- **Chain-of-Thought 几乎是必须的**（复杂任务）
- **Few-shot examples 效果很明显**（格式类任务）
- **约束条件要写更明确**（老模型更容易跑题）
- **一次只做一件事**（老模型多任务能力弱）
- **输出要短**（长输出质量下降更明显）
