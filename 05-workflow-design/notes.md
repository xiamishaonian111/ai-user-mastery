# 工作流设计 Workflow Design

> 目标：能把任何复杂任务拆解成 AI 能高质量完成的步骤

---

## 四个核心原则

**1. 单任务原则 Single-task principle**
不要让 AI 同时做太多事。"重构模块、加测试、写文档"效果远不如分三次说。

**2. 验证点设计 Checkpoint design**
在关键步骤后主动检查，不要等到最后才发现跑偏。对话越长，AI 越容易漂移。

**3. 人做判断，AI 做执行 Human judgment, AI execution**
永远不要把判断权交给 AI：
| 你来做 | AI 来做 |
|-------|--------|
| 方向值不值得做 | 把方向写成文档 |
| 设计合不合理 | 把设计实现出来 |
| 结果够不够好 | 根据反馈修改 |

**4. 迭代而非一次完成 Iterative over one-shot**
先要草稿 → 你看一遍 → 给具体反馈 → 再迭代

## 多 Agent 工作流 Multi-agent Workflow

### 什么时候需要多 Agent

| 场景 | 原因 |
|------|------|
| 任务太长，单个对话撑不住 | 各自有干净的 context |
| 任务可以并行 | 多个 agent 同时跑，节省时间 |
| 不同部分需要不同专业视角 | 每个 agent 有自己的 Role |
| 需要互相验证 | Agent A 做，Agent B 检查 |

### 设计三步骤

1. **拆任务** — 哪些步骤可以独立？哪些有依赖关系？
2. **定角色** — 每个 agent 的 Role 是什么，Context 给什么
3. **设验证点** — 哪些节点人工介入，哪些放行

---

## Orchestrator 指令模板

```markdown
# 任务：[任务名称]

## Agents 和职责
- Agent 1：[职责描述]
- Agent 2：[职责描述]
- Agent 3：[职责描述]

## 执行顺序和依赖
1. 启动 Agent 1
2. ⚠️ 暂停，等待人工确认
3. 确认后启动 Agent 2
4. ⚠️ 暂停，等待人工确认
5. 确认后同时启动 Agent 3 和 Agent 4（并行）
6. 两者完成后汇总成最终输出

## 人工介入规则
- 每次暂停，把当前输出展示给用户
- 用户输入"ok"或"继续" → 进入下一步
- 用户输入其他内容 → 视为修改意见，当前 agent 重做，再次暂停等待确认

## 输出格式
每个 agent 完成后输出：
[agent名称] 完成
[输出内容]
等待确认...
```

## 多 Agent 互相验证

**核心：** 不要让同一个 Agent 又做又检查，让另一个 Agent 专门挑错。

### 基本模式

```
Agent A：执行任务，给出结果
    ↓
Agent B：专门找问题（不同视角）
    ↓
问题 ≥ 3 个 → 传回给 Agent A 修改 → 再走 Agent B
问题 < 3 个 → 停止，等待人工确认
```

### Agent B 的 Role 怎么设计

```
[Role] 你是一个严格的 reviewer，专门找逻辑漏洞和错误假设。
[Task] 找出所有假设是否成立、逻辑是否有漏洞、有没有遗漏重要因素。
不要给建设性建议，只指出问题。
```

### Orchestrator 指令写法

```markdown
你是 Orchestrator，负责协调以下工作：

1. 先作为 Agent A，完成分析，把结果存下来
2. 再切换成 Agent B，拿 Agent A 的结果来挑问题
3. 如果发现 3 个以上问题：
   → 切换回 Agent A，根据问题修改
   → 再切换成 Agent B 重新检查
   → 最多循环 3 次
4. 问题少于 3 个或达到循环上限：停下来等我确认
```

**重要：必须设循环上限**，否则 A 和 B 可能无限循环。超过上限就停下来让人介入。

### 注意

Orchestrator 不是一个程序，就是你写的规则。Claude 既是 Orchestrator 又是各个 Agent，靠你的指令切换角色。规则写得越清楚，执行越稳定。

## Best Practices

**1. 明确写"暂停"，不要依赖 AI 自己判断**
写"⚠️ 暂停，等待人工确认"比"需要时询问用户"效果好——前者是硬性指令，后者 AI 可能跳过。

**2. 明确定义用户输入的含义**
```
用户输入"ok"或"继续" → 进入下一步
用户输入其他内容 → 视为修改意见，重做
```

**3. 每个 agent 的 context 单独控制**
每个 agent 启动时只给它需要的信息：
- 上一个 agent 的输出结果
- 自己的任务描述
- 不需要知道其他 agent 的存在

**4. 验证点选在高风险节点**
如果这里出错，会影响后面所有工作的地方，必须放验证点。

## 示例：PRD 多 Agent 工作流

```
Agent 1（市场分析）
    ↓
⚠️ 人工检查
    ↓
Agent 2（功能设计）
    ↓
⚠️ 人工检查
    ↙        ↘
Agent 3      Agent 4
（技术约束） （上线计划）
    ↘        ↙
    Orchestrator
    （汇总成 PRD）
```

Agent 3 和 4 并行，因为都只依赖 Agent 2，互相之间没有依赖。
