# 成为优秀 AI User 的学习路线

> 创建于 2026-04-11
> 工具：Claude Code 及同类 Agentic AI 工具

---

## 核心前提

模型越来越强，传统"提示词技巧"的价值在下降。
真正值得学的，是**模型再强也解决不了的问题**，以及**把 AI 能力用满的方法**。

---

## 阶段一：建立正确的心智模型（1-2 周）

> 目标：知道 AI 能做什么、在哪里会失败、为什么

### 核心概念

**1. Context Window 是一切的根源**
- AI 只能看到 context window 里的内容，看不到就不知道
- 所有"AI 不理解我"的问题，90% 是 context 问题，不是 prompt 问题
- 这是后面所有技能的底层原理

**2. AI 的可预测失败模式**
即使是最强的模型，也会在以下情况可预测地失败：
- 任务太模糊（目标不清晰）
- 需要的信息不在 context 里（知识缺口）
- 任务太长导致"迷失"（注意力漂移）
- 需要精确计算或严格逻辑（数学、复杂推理）
- 需要最新信息（知识截止日期）

**3. "够强的模型"改变了什么，没改变什么**
- 改变了：不再需要繁琐的提示词格式和技巧
- 没改变：AI 依然需要清晰的目标和足够的上下文
- 新问题：模型太强容易让你放松警惕，在它悄悄出错的地方忽视

### 实践
- 刻意让 Claude 做一件它不该做好的事（比如精确计数、实时信息），观察它如何失败
- 记录一周内 AI 给你不满意结果的所有案例，分析根本原因

---

## 阶段二：上下文工程（2-3 周）

> 目标：让 AI 在你的项目里始终"知道"它需要知道的一切

这是目前**最被低估、价值最高**的技能。提示词技巧会随模型进化而过时，上下文工程不会。

### 核心技能

**1. CLAUDE.md —— 给 AI 的项目说明书**
- 每个项目根目录放一个 CLAUDE.md
- 包含：项目是什么、技术栈、代码规范、不要做什么
- AI 每次启动都会读它，等于给了 AI 一个永久记忆
- 写得越好，AI 在你项目里的表现越稳定

**2. 结构化你的信息**
- 文件名、目录结构、注释质量直接影响 AI 的理解
- "AI 能不能快速找到相关信息"是设计项目结构的新维度
- 好的文档不只是给人看的，也是给 AI 的接口

**3. Memory 系统设计**
- 哪些信息应该放在 CLAUDE.md（永久，所有对话都需要）
- 哪些信息应该在对话开头手动补充（临时，这次对话特有）
- 哪些信息让 AI 自己从文件里找（按需，不需要提前塞入）

**4. 信息密度控制**
- context 不是越多越好，无关信息会"稀释"有用信息
- 学会判断什么该放、什么该删

### 实践
- 给你目前最常用的项目写一个 CLAUDE.md，观察 AI 行为变化
- 对比有 CLAUDE.md 和没有时，做同样任务的差距

## 阶段三：Claude Code 功能深度使用（3-4 周）

> 目标：把 Claude Code 用到 80%，而不是 20%

大多数人只用到了 Claude Code 的基础对话功能，以下功能几乎没人用：

### Hooks —— 自动化你的工作流
- 在特定事件触发时自动执行命令
- 例：每次 Claude 写完文件，自动运行格式化；每次对话开始，自动加载环境信息
- 适合：重复性的、每次都要做的操作

### MCP Servers —— 给 Claude 装插件
- 让 Claude 能操作你的数据库、API、浏览器、日历……
- 社区有大量现成的 MCP server（GitHub、Notion、Slack 等）
- 适合：需要 Claude 访问外部系统的场景

### 自定义 Skills（斜杠命令）
- 把常用的复杂操作封装成一个命令
- 例：`/review-pr` 自动拉取 PR、分析改动、给出意见
- 适合：固定的、有标准流程的任务

### 权限与安全设置
- 理解不同权限模式的含义（auto/plan/acceptEdits）
- 在不同场景下选择合适的权限级别

### 实践
- 为自己的日常工作流写一个 Hook
- 安装 2-3 个 MCP server（从自己最常用的工具开始）
- 把最常做的一个复杂操作封装成 Skill

## 阶段四：任务分解与工作流设计（持续）

> 目标：能把任何复杂任务拆解成 AI 能高质量完成的步骤

这是从"会用 AI"到"用好 AI"最关键的跨越。

### 核心原则

**1. 单任务原则**
不要让 AI 同时做太多事。"帮我重构这个模块、加测试、写文档"效果远不如分三次说。

**2. 验证点设计**
在关键步骤后主动检查，而不是等到最后才发现方向跑偏。
对话越长，AI 越容易漂移——定期"重新对焦"。

**3. 人做判断，AI 做执行**
分清楚哪些决策必须自己做（方向、取舍、标准），哪些执行可以完全交给 AI。

**4. 迭代而非一次完成**
复杂任务不要期待一次就完美。先让 AI 给一个草稿，反馈，再迭代。

### 实践
- 拿一个你觉得"AI 做不好"的复杂任务，把它分解成小步骤重试
- 记录分解前后的质量差距

---

## 阶段五：评估与校准（持续）

> 目标：知道什么时候信任 AI，什么时候必须自己验证

### 核心技能

**1. 输出质量判断**
- 对自己领域的内容，能快速判断 AI 输出是否合理
- 对不熟悉的领域，知道如何验证（让 AI 解释推理过程，找外部来源）

**2. 错误模式识别**
- 自信但错误（hallucination）：AI 说得很确定但是假的
- 方向正确但细节错误：大框架对，但有小 bug 或细节偏差
- 过度保守：能做但不敢做，反复确认

**3. 反馈质量**
- 模糊反馈（"不对，再来一次"）效果很差
- 具体反馈（"第二段的逻辑有问题，因为……"）效果很好
- 学会给 AI 好的反馈，和教一个新员工一样

## 学习节奏建议

```
阶段一（心智模型）  ████░░░░░░  1-2 周，以观察和思考为主
阶段二（上下文工程）████████░░  2-3 周，最重要，花时间在实践上
阶段三（工具深度）  ██████░░░░  3-4 周，边用边学
阶段四（工作流）    持续进行，每次复杂项目都是练习机会
阶段五（评估）      持续进行，融入日常使用习惯
```

**最重要的习惯：**
每次 AI 给出不满意的结果，停下来分析根本原因——是 context 不够？任务太模糊？还是这本来就是 AI 做不好的事？这些分析比看任何教程都有价值。

## 关于那些你听说过的概念

- **提示词工程**：基础仍然有用，但不需要深钻，模型够强了
- **上下文工程**：⭐ 最值得学，且会越来越重要
- **RAG**：作为 AI Builder 你已经学过了；作为 User，知道概念即可
- **Agentic workflow**：即 Claude Code 的核心使用模式，阶段三会覆盖
- **Fine-tuning**：作为 User 几乎用不到，跳过
