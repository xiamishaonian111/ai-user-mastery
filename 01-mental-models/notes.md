# 心智模型 Mental Models

> 目标：知道 AI 能做什么、在哪里会失败、为什么

---

## 一、AI 的可预测失败模式

即使最强的模型，也会在以下情况可预测地失败：

| 失败模式 | 现象 | 解法 |
|---------|------|------|
| **目标模糊** | AI 会猜你的意图，给看起来合理但不对的答案 | 说清楚目标和约束 |
| **信息不在 context** | AI 会自信地编造，而不是说"我不知道" | 把必要信息放进 context |
| **对话太长，注意力漂移** | 早期约定被悄悄违反 | 重要约束放 CLAUDE.md，定期重新对焦 |
| **精确计算/严格逻辑** | 可能算错，Agentic 系统有工具可缓解 | 让它用工具算，或自己验证 |
| **需要最新信息** | 知识有截止日期，会说旧信息 | 接入搜索工具，或自己提供信息 |

**注意：** 精确计算和最新信息这两条，在有工具的 Agentic 系统里已大幅改善，但模型仍可能跳过工具直接回答——要警惕。

**减少幻觉的三个方向：**
1. 给它更多正确信息（context 工程）
2. 让它暴露推理过程（Chain-of-Thought）
3. 设计验证环节（工作流设计）

## 二、AI 的注意力机制

### Lost in the Middle

**出处：** Liu et al. (2023) 论文《Lost in the Middle: How Language Models Use Long Contexts》
（发表于 ACL 2024，arXiv: 2307.03172）

**核心发现：**
模型对上下文的不同位置有明显的注意力偏差——
- **开头**（primacy effect）：记忆最强
- **结尾**（recency effect）：记忆次强
- **中间**：最容易被忽略，即使信息明确在那里

**实验验证：**
论文把答案藏在长文档的不同位置，测试模型能否找到。答案在开头/结尾时准确率高，在中间时准确率显著下降。

**对你的影响：**
- 把最重要的约束条件放在 prompt 的开头或结尾，不要埋在中间
- CLAUDE.md 里的关键规则放在文件开头
- 长对话中的重要设定容易随时间漂移（因为滚入了"中间"位置）

---

### Semantic Dilution（语义稀释）

**出处：** 这是一个描述性术语，不是某篇特定论文的命题，而是实践者对 LLM 行为的观察总结。多个研究和工程博客讨论过这个现象（包括 Anthropic、OpenAI 的工程博客），但没有单一权威出处。

**核心现象：**
当有效信息被大量无关内容稀释时，模型对这些信息的权重会下降——即使信息本身没有被删除。

**两种主要表现：**

1. **信息位置稀释**：重要约束埋在长段落中间，被周围文字"淹没"
   - 把 "不要用 emoji" 写在一大段背景介绍的中间 → 模型可能忽略
   - 把它单独列出 → 效果明显更好

2. **格式稀释**：过度使用 markdown（大量 `**bold**`、标题、列表）反而让模型难以判断哪里是真正重要的
   - 所有东西都加粗 = 没有东西加粗
   - 适度格式化，让视觉权重和语义权重一致

**和 Lost in the Middle 的关系：**
两者相关但不同：
- Lost in the Middle = 位置问题（中间位置系统性被忽略）
- Semantic Dilution = 密度问题（有效信息被无关内容稀释，可以发生在任何位置）

## 二、Claude Code 的输入结构

### 完整结构图

每次对话，模型收到的完整输入是：

```
┌─────────────────────────────────────┐
│           System Prompt             │  ← 权重最高，全程有效
│  - Claude Code 内置指令              │
│  - CLAUDE.md 内容（如有）            │
│  - Memory 文件内容（如有）            │
└─────────────────────────────────────┘
             ↓
┌─────────────────────────────────────┐
│           对话历史                   │  ← 会越来越长
│  - 之前的 user/assistant 轮次        │
│  - 工具调用结果（读文件、执行命令等）  │
│  - 压缩后变成摘要                    │
└─────────────────────────────────────┘
             ↓
┌─────────────────────────────────────┐
│         当前 User Message           │  ← 你刚发的这条
└─────────────────────────────────────┘
```

### 关键细节

**CLAUDE.md 在哪里：**
- 在 System Prompt 里，每次对话自动加载
- 范围：当前项目（目录下有 CLAUDE.md 的项目）

**Memory 文件在哪里：**
- 也在 System Prompt 里（`~/.claude/projects/.../memory/`）
- 范围：全局，所有项目的对话都会加载

**工具调用（读文件等）在哪里：**
- 在对话历史里，以 tool result 的形式存在
- 模型主动决定要读哪些文件，读完结果进入对话历史

**压缩（Compaction）做了什么：**
- 只压缩对话历史，System Prompt 不变
- 压缩后：对话历史变成一段摘要，放在对话历史开头
- System Prompt 内容（CLAUDE.md、memory）每次重新加载，不依赖压缩保留

### Index 文件的工作方式（迭代读取）

CLAUDE.md 只存路标，不预加载内容：

```
1. CLAUDE.md 说："完整结构见 /docs/index.md"
2. 模型看到这个路标
3. 用户问问题 → 模型决定 "我需要去读 index 文件"
4. 模型发出 Read("/docs/index.md") 工具调用
5. 工具返回内容 → 进入对话历史
6. 模型现在才真正知道 index 内容
7. 模型可能继续读更多文件...
8. 信息够了 → 给用户回答
```

这是 AI Agent 的核心工作方式：模型是主动驾驶员，自己决定要读什么、什么时候读。

## 三、如何定位 AI 的角色

### 不是工具，不是同事——是"刚入职的天才新员工"

这是目前最有用的心智模型。

**错误的定位：**

| 定位 | 问题 |
|------|------|
| 当工具用（像用锤子） | 你做所有思考，只让它执行，严重低估了它的能力 |
| 当同事用（平等协作） | 过度信任，它没有你的上下文、没有利益相关，容易出错你不察觉 |
| 当搜索引擎用 | 只问问题，不让它做事，浪费了它最强的能力 |

**正确的定位：你是 team lead，AI 是刚入职的天才新员工**

这个新员工：
- 能力极强，学什么都快，执行力超强
- 但对你的公司、你的项目、你的偏好**一无所知**
- 需要你交代清楚背景，才能做好事情
- 会自信地做错事，而不是承认自己不知道
- 不会主动问"这样做对吗"，你要主动设验证点

**这意味着你的工作是：**
- 交代清楚背景和目标（上下文工程）
- 把复杂任务拆成小块（任务分解）
- 在关键节点检查输出（验证点设计）
- 给具体反馈，而不是"再来一次"

**你不需要做的：**
- 手把手写每一步怎么执行
- 担心它"不懂"代码/写作/分析——它懂
- 反复重复同样的指令——说一次，说清楚

---

### AI 没有记忆，只有 context

每次新对话，AI 对你是完全陌生的。

它不记得你上次说过什么、你喜欢什么风格、你的项目是什么。
它只知道**这次对话里你给了它什么**。

这就是为什么 CLAUDE.md 和 Memory 文件很重要——它们是你给 AI 的"永久记忆替代品"。

## 四、AI 是概率机器，不是确定性工具

同样的输入，不同的输出。这不是 bug，是设计。

**实际影响：**
- 同一个问题问两次，可能得到两个不同的答案
- 答案"看起来确定"≠ 答案是对的
- 它说"根据我的知识"不代表它在验证，它只是在生成

**你需要的习惯：**
- 对关键输出，不要只看第一次结果
- 对它不擅长的领域（数字、逻辑、事实），主动验证
- 让它解释推理过程，而不只是给结论——推理过程有没有漏洞，你一眼能看出来

## 五、什么时候信任，什么时候验证

不是所有输出都需要验证，但你要知道边界在哪。

| 场景 | 信任程度 | 原因 |
|------|---------|------|
| 你熟悉的领域 | 高 | 你能快速判断对错 |
| 格式类任务（改写、翻译、排版） | 高 | 容易肉眼检查 |
| 代码逻辑 | 中 | 看起来对但可能有 bug，跑一遍 |
| 事实性声明（数字、日期、引用） | 低 | 容易自信地说错 |
| 你完全不懂的领域 | 低 | 你没有能力判断对错 |

**核心原则：**
> 你对一个领域越不懂，越不能只靠 AI——因为你没有能力发现它在哪里错了。

## 六、"够强的模型"改变了什么，没改变什么

**改变了：**
- 不需要繁琐的提示词格式（说人话就够了）
- 不需要手把手解释每一步怎么执行
- 复杂任务的执行质量大幅提升

**没改变：**
- AI 依然需要清晰的目标和足够的上下文
- AI 依然会在信息不足时自信地编造
- AI 依然没有你的判断力——知道"做什么"和"为什么"

**新风险（强模型特有）：**
模型越强，输出越"看起来正确"，你越容易放松警惕。
它悄悄出错的地方，你反而更难发现。

## 七、Memory 系统设计

### Memory 是什么

Claude Code 的 memory 就是一堆 markdown 文件，存在 `~/.claude/projects/.../memory/`。每次对话开始，这些文件整个塞进 System Prompt。没有短期/中期/长期的分层机制，就是文件→System Prompt。

### Memory 的类型分工

| 类型 | 存什么 | 作用范围 |
|------|-------|---------|
| user | 你是谁、偏好、背景 | 所有项目 |
| feedback | AI 行为纠正记录 | 所有项目 |
| project | 某项目进度、决策 | 理论上项目级，实际全局 |
| reference | 外部系统位置 | 按需 |

### 关键局限

- Memory 文件**没有自动清理机制**，靠人工管理
- project 类型的 memory **实际上是全局的**，会出现在所有项目的对话里
- 项目结束后必须手动清理 project memory，否则污染后续对话

### Best Practice

```
全局 memory 文件   → 只放真正跨项目通用的东西（偏好、反馈）
项目的 CLAUDE.md  → 放项目特有的东西（背景、进度、规范）
```

project 类型的 memory 内容，实际上应该写进项目的 CLAUDE.md，不要放全局 memory。

---

## 八、CLAUDE.md 的层级加载

Claude Code 从当前目录一路向上找所有 CLAUDE.md，全部加载进 System Prompt：

```
~/CLAUDE.md                        ← 全局个人偏好
~/项目名/CLAUDE.md                 ← 项目规范和背景
~/项目名/子模块/CLAUDE.md           ← 子模块细节（通常不需要）
```

**分层设计原则：**
- 越靠上：越通用，适用所有项目
- 越靠下：越具体，只对这个模块有效
- 实际上大多数情况只需要两层：`~/CLAUDE.md` + 项目根目录的 `CLAUDE.md`

## 九、Chain-of-Thought（让 AI 暴露推理过程）

**核心：不要让它直接给结论，让它先说它怎么想的。**

为什么有用：AI 生成文字是线性的，写下来的思考过程会影响后面的结论。强迫它先写推理，等于强迫它更认真地想。同时你能看到它在哪一步出了问题。

### 三种用法

**1. 直接要求推理步骤**
```
请按以下步骤分析，每步都写出来：
1. 支持的理由
2. 反对的理由
3. 关键假设
4. 基于以上，给出结论
```

**2. 让它列出假设和来源**
```
你刚才的建议基于哪些数据和假设？
请逐一列出，并注明：
- 如果是事实数据：给出来源
- 如果是推理出来的假设：说明你的推理过程
```

**3. 让它自我质疑**
```
请指出这份分析最可能在哪里出错。
按影响程度排列：哪些错误会让结论失效？哪些只是细节偏差？
```

### 什么时候用

| 场景 | 用不用 |
|------|-------|
| 复杂分析、推理类任务 | 用 |
| 你不熟悉的领域 | 一定用 |
| 简单格式类任务（改写、翻译） | 不需要 |

## 十、实践要点

基于以上原理，对你日常使用的影响：

| 原理 | 实践建议 |
|------|---------|
| Lost in the Middle | 重要约束放开头或结尾，不要埋中间 |
| Semantic Dilution | 关键信息单独一行/一段，不要淹在大段文字里 |
| System Prompt 权重最高 | CLAUDE.md 里的规定最稳定，对话里说的可能漂移 |
| 对话越长越容易漂移 | 长对话定期"重新对焦"，重申关键约束 |
| 模型迭代读文件 | CLAUDE.md 写清楚"去哪找什么"，帮模型快速导航 |
